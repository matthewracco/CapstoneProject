TL;DR - What Is This Project?

We're building a Locker Rental System — think of it like renting a locker at an airport, gym, or school, but digitally.

  • Customers scan a QR code on a physical locker, rent it through their phone/browser, get an access code to unlock it.
  • Staff manage lockers, handle overdue rentals, override locks.
  • Business Owners configure pricing, manage staff, view analytics.
  • Platform Admins onboard new businesses (tenants).

It's multi-tenant — one platform serves many businesses, each with their own lockers and users.

The 4 User Roles

Customer

Regular user who rents lockers

  • Register & login
  • Browse available lockers
  • Scan QR code on physical locker
  • Rent a locker
  • Get access code to unlock
  • Extend rental
  • End rental
  • View rental history
  • Receive notifications

Staff

Employee managing day-to-day operations

  • Everything a customer can do, plus:
  • View ALL lockers and their status
  • Override locker status (unlock, maintenance)
  • View overdue rentals
  • Assign lockers to specific users
  • Approve/deny user access requests

Business Owner

Owns/runs the facility

  • Everything staff can do, plus:
  • Configure facility settings
  • Set pricing per locker size
  • Add/remove/decommission lockers
  • Manage staff accounts
  • View analytics & reports
  • Configure rental policies

Platform Admin

Manages the whole platform

  • Onboard new tenants (businesses)
  • Monitor system health
  • View platform-wide metrics
  • (Low priority for demo)

Flow: Rental Lifecycle (Backend)

What happens in the database when a rental is created and completed.

sequenceDiagram
    actor Customer
    participant Frontend
    participant API as Express API
    participant Service as Rental Service
    participant DB as SQLite (Prisma)

    Customer->>Frontend: Click "Rent Locker A-001"
    Frontend->>API: POST /api/v1/rentals {lockerId}
    API->>API: Auth middleware (verify JWT)
    API->>API: Tenant middleware (extract tenantId)
    API->>API: Zod validation

    API->>Service: createRental(tenantId, userId, lockerId)
    Service->>DB: BEGIN TRANSACTION
    Service->>DB: Find locker WHERE id = lockerId AND status = AVAILABLE
    alt Locker is available
        Service->>DB: UPDATE Locker SET status = OCCUPIED
        Service->>DB: CREATE Rental (status=ACTIVE, cost=price)
        Service->>DB: COMMIT
        Service-->>API: Return rental object
        API-->>Frontend: 201 Created {rental}
        Frontend-->>Customer: Show confirmation + rental code
    else Locker not available
        Service->>DB: ROLLBACK
        Service-->>API: Throw error
        API-->>Frontend: 409 Conflict "Locker not available"
        Frontend-->>Customer: Show error message
    end

    Note over Customer,DB: Later... customer ends rental

    Customer->>Frontend: Click "End Rental"
    Frontend->>API: POST /api/v1/rentals/:id/complete
    API->>Service: completeRental(rentalId)
    Service->>DB: BEGIN TRANSACTION
    Service->>DB: UPDATE Rental SET status=COMPLETED, endTime=now
    Service->>DB: UPDATE Locker SET status=AVAILABLE
    Service->>DB: COMMIT
    API-->>Frontend: 200 OK
    Frontend-->>Customer: Rental ended!

stateDiagram-v2

    OCCUPIED --> AVAILABLE : Rental completed/ended
    OCCUPIED --> MAINTENANCE : Staff marks for repair

    AVAILABLE --> MAINTENANCE : Staff marks for repair

    AVAILABLE --> ASSIGNED : Owner assigns to user

    note right of OCCUPIED : Locker is actively rented\nCannot be rented by another user

How it works in code:

  • Every database row has a tenantId column
  • When a user logs in, their JWT contains their tenantId
  • The clerkTenant.js middleware extracts it and adds it to req.tenantId
  • Every database query filters by WHERE tenantId = req.tenantId
  • Tenant A can NEVER see Tenant B's data
